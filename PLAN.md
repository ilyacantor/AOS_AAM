# Runner Architecture Implementation Plan

## Context

AAM currently operates as a stateless API: humans/AOD push data in, AAM stores + infers,
operators export to DCL. For 500 production pipes, we need **AAM to dispatch ephemeral
workers** that fetch source data and push it into DCL — without AAM ever touching the
payload bytes.

**What exists today:**
- Adapter pattern (4 fabric plane adapters with connect/discover/health)
- Collector runs with status tracking (running → completed/failed)
- Provenance model (discovered_by, lineage_hints, aod_run_id)
- DCL export (GET declared-pipes, POST push with payload recording)
- Candidate → Pipe inference pipeline
- PII redaction on observations

**What's missing:**
- Job Manifest model (the "instruction packet" for a Runner)
- Runner dispatcher (create manifest → queue → track)
- DCL ingestion endpoint (POST /api/dcl/ingest — Runners push here)
- Schedule triggers (cron-like dispatch)
- FARM verification hook (post-ingest validation)

## Phase 1: Job Manifest Model + Runner Dispatch

### 1a. Job Manifest Model (`app/models.py`)

New Pydantic models:

```python
class SourceSpec(BaseModel):
    pipe_id: str
    system: str                         # vendor name
    adapter: str                        # rest_api | jdbc | kafka | ipaas | webhook
    endpoint_ref: dict                  # opaque connection info
    credentials_ref: Optional[str]      # vault://aam/secrets/pipe_xxx (reference only)
    query: Optional[str]                # extraction query/filter

class TransformSpec(BaseModel):
    schema_map: dict                    # source_field → {target, unit, scale, dimension}
    grain: Optional[str]               # quarter | month | day
    period_field: Optional[str]
    period_format: Optional[str]

class TargetSpec(BaseModel):
    dcl_url: str                        # POST endpoint
    auth_token_ref: Optional[str]       # vault reference
    tenant_id: Optional[str]
    snapshot_name: Optional[str]

class RunLimits(BaseModel):
    max_rows: int = 100_000
    timeout_seconds: int = 300
    retry_count: int = 2

class JobManifest(BaseModel):
    manifest_version: str = "1.0"
    run_id: str                         # generated by AAM
    source: SourceSpec
    transform: Optional[TransformSpec]
    target: TargetSpec
    provenance: dict                    # run_timestamp, triggered_by
    limits: RunLimits = RunLimits()
    farm_verification: bool = False
```

### 1b. Runner State Machine (`app/db/runner_jobs.py`)

New table `runner_jobs` extending the pattern of `collector_runs`:

| Column | Type | Description |
|--------|------|-------------|
| job_id | TEXT PK | = run_id from manifest |
| pipe_id | TEXT | pipe being collected |
| status | TEXT | queued → dispatched → running → pushing → completed / failed |
| manifest | JSON | full JobManifest |
| dispatched_at | TEXT | timestamp |
| started_at | TEXT | runner reports start |
| completed_at | TEXT | runner reports end |
| rows_transferred | INT | count pushed to DCL |
| error_message | TEXT | if failed |
| dcl_response | JSON | response from DCL ingest |

Functions:
- `create_runner_job(manifest: JobManifest) → job_id`
- `update_runner_status(job_id, status, **kwargs)`
- `get_runner_job(job_id) → dict`
- `list_runner_jobs(pipe_id=None, status=None, limit=50) → List[dict]`

### 1c. Runner Dispatcher Service (`app/services/runner_dispatch.py`)

```
build_manifest(pipe: DeclaredPipe, trigger: str) → JobManifest
    - Reads pipe fields (endpoint_ref, source_system, fabric_plane)
    - Generates run_id: f"run_{date}_{pipe.source_system}_{seq:03d}"
    - Resolves adapter type from pipe.fabric_plane + pipe.transport_kind
    - Sets target.dcl_url from config
    - Returns manifest

dispatch_job(manifest: JobManifest) → job_id
    - Stores job in runner_jobs table (status=queued)
    - In v1: executes inline via adapter (async)
    - In v2: enqueues to background worker pool
    - Returns job_id

dispatch_pipe(pipe_id: str, trigger: str = "manual") → job_id
    - Loads pipe → builds manifest → dispatches
```

### 1d. Runner Endpoints (`app/routers/runners.py`)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/api/runners/dispatch` | Dispatch single pipe (body: {pipe_id, trigger?}) |
| POST | `/api/runners/dispatch-batch` | Dispatch multiple pipes (body: {pipe_ids?, filter?, trigger}) |
| GET | `/api/runners/jobs` | List jobs (query: pipe_id, status, limit) |
| GET | `/api/runners/jobs/{job_id}` | Job details + manifest |
| POST | `/api/runners/jobs/{job_id}/status` | Runner reports status update (callback) |
| GET | `/api/runners/jobs/{job_id}/manifest` | Download manifest JSON |

## Phase 2: DCL Ingestion Endpoint

### 2a. Ingest Endpoint (`app/routers/dcl_ingest.py`)

```
POST /api/dcl/ingest
Headers:
    x-run-id: run_20260215_sf_revenue_001
    x-api-key: (optional, for auth)
Body:
    {
        "meta": {
            "source_system": "salesforce",
            "pipe_id": "pipe_sf_01",
            "run_timestamp": "2026-02-15T10:00:00Z",
            "schema_version": "1.2",
            "row_count": 2
        },
        "data": [
            {"entity_id": "opp_001", "metric_revenue": 50000, ...},
            ...
        ]
    }
```

Response: `{ "status": "accepted", "run_id": "...", "rows_stored": N }`

Logic:
- Validate run_id exists in runner_jobs (authorized run)
- Validate meta.pipe_id matches job manifest
- Store payload (dcl_ingested_payloads table or append to existing dcl_pushes)
- Update runner_job status → "completed"
- Tag all rows with run_id + pipe_id + timestamp for provenance
- If manifest.farm_verification: enqueue FARM check

### 2b. Ingested Data Store (`app/db/dcl_ingest.py`)

New table `dcl_ingested` (mirrors dcl_pushes but for inbound):

| Column | Type | Description |
|--------|------|-------------|
| ingest_id | TEXT PK | UUID |
| run_id | TEXT | from manifest |
| pipe_id | TEXT | pipe that produced this data |
| source_system | TEXT | vendor |
| row_count | INT | rows received |
| payload_hash | TEXT | SHA-256 for dedup |
| payload | JSON | full data payload |
| ingested_at | TEXT | timestamp |
| schema_version | TEXT | from meta |

## Phase 3: Schedule Triggers

### 3a. Pipe Schedule Field

Add optional `schedule` to pipe model:

```python
class PipeSchedule(BaseModel):
    cron: Optional[str]          # "0 6 * * *" = daily 6am
    interval_minutes: Optional[int]
    enabled: bool = True
```

### 3b. Scheduler Service (`app/services/scheduler.py`)

Lightweight in-process scheduler (APScheduler or simple asyncio loop):

```
on startup:
    load all pipes with schedule.enabled = True
    register cron jobs → dispatch_pipe(pipe_id, trigger="cron:{expr}")

on pipe update:
    re-register schedule if changed

tick loop (every 60s):
    check for due pipes → dispatch
```

For v1, keep it simple: a single async background task that polls
pipe schedules every 60s. No external dependencies.

### 3c. Schedule Endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| PUT | `/api/pipes/{id}/schedule` | Set/update schedule |
| DELETE | `/api/pipes/{id}/schedule` | Disable schedule |
| GET | `/api/runners/schedule` | List all scheduled pipes + next run |

## RACI Refinements (Post-Validation)

Three hardening requirements from RACI review:

### Refinement A: Runner Heartbeat / Dead-Letter (RACI Row 48)

Risk: Runner crashes silently → AAM thinks job is "running" forever.
Fix: `PUT /api/runners/callback/{run_id}` — Runner MUST call on success or failure.
Stale job reaper: background sweep marks jobs "running" for >timeout_seconds as `timed_out`.
Added to runner_jobs: `last_heartbeat` column (TEXT timestamp).

### Refinement B: Secret-Reference-Only Manifests (RACI Row 82)

Risk: Raw credentials persisted in runner_jobs.manifest JSON or logs.
Fix: JobManifest.source.credentials_ref and target.auth_token_ref are ONLY vault URIs
(`vault://aam/secrets/xxx`). Runner resolves at runtime via env var or vault API.
The manifest stored in DB never contains plaintext secrets.

### Refinement C: x-schema-hash Header on Ingest (RACI Row 63)

Risk: Source schema changes silently between runs → stale data in DCL.
Fix: Runner computes SHA-256 of the data structure (sorted field names + types)
and sends as `x-schema-hash` header. DCL compares to previous run for same pipe.
If mismatch → stores the data but flags a "schema_drift" event.
Added to dcl_ingested: `schema_hash` column (TEXT).

## Key Design Decisions

1. **v1 = inline execution** — Runners execute in-process (asyncio tasks)
   before graduating to Docker/serverless. This keeps the prototype
   deployable as a single process while proving the manifest/dispatch model.

2. **Manifest is immutable** — once dispatched, the manifest is frozen in
   the runner_jobs table. No in-flight modifications.

3. **Credentials are references** — `vault://aam/secrets/xxx` strings only.
   Runner resolves at runtime. Manifest in DB never has plaintext secrets.

4. **DCL ingest validates run_id** — only data from authorized runner jobs
   is accepted. No open ingest endpoint.

5. **FARM verification is opt-in** — per-manifest flag. Default false.
   Keeps the happy path fast.

6. **Schema normalization at Runner** — Option A from the discussion.
   The transform.schema_map in the manifest tells the Runner how to
   map source fields to DCL semantic fields before pushing.

7. **Schema drift detection at ingest** — x-schema-hash header compared
   to previous run. Mismatch → drift event flagged, data still accepted.

8. **Dead-letter reaper** — Jobs stuck in "running" past timeout_seconds
   are marked "timed_out" by a periodic sweep.
